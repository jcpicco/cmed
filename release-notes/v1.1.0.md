# Release Notes 

Version 1.1.0 introduces significant improvements to the system architecture, user experience optimizations, a substantial evolution of the PDF export system, and the addition of a new independent patient notes module.

---

# Backend

## New Features

### Patient Notes System

A complete system of independent notes directly associated with the patient profile has been introduced, decoupled from the medical record.

- Dedicated REST endpoints:  
  `/api/patients/{id}/notes`
- Full CRUD operations.
- Pagination support.
- Allows coexistence of notes linked to medical records and notes linked exclusively to the patient.

### New Field in MedicalRecord

A new nullable field named `background` has been added to the `MedicalRecord` entity.

- Allows storing general medical background information for the patient.
- Backward compatible with existing records (not mandatory).
- Fully integrated into medical record persistence and retrieval flows.

### Interactive API Documentation

Interactive API documentation is now available via Swagger UI.

- Available locally at:  
  `http://localhost:8080/swagger-ui/index.html`
- Enables endpoint exploration, model inspection, and direct API testing.

### Portable Support (Executables)

Build scripts have been added to generate self-contained executable artifacts:

- `build_portable.ps1`
- `build_portable.sh`

These scripts package the application into a single executable (JAR/EXE) using the Spring `portable` profile, enabling simplified deployments through direct execution.

---

## Improvements and Refactoring

### REST Route Standardization

Endpoints have been refactored to align with REST sub-resource conventions and naming best practices:

- `medical-records/{id}/notes`
- `patients/{id}/notes`

Additionally, all endpoints have been migrated from **camelCase** to **kebab-case**, improving URL readability, consistency, and adherence to REST conventions.

### Project Structure Simplification

The internal `app/` folder has been removed, and the Spring Boot engine has been moved directly to the root of the `cmed-app/` module, simplifying the overall project structure.

### Docker Decoupling

The following have been removed:

- `Dockerfile`
- `docker-compose.yml`
- Heavy deployment configurations

Deployment logistics are now unified through the new portable build scripts.

---

## Bug Fixes

### Database – Notes

Fixed note insertion to allow null values in the `medical_record_id` column when the note is associated exclusively with the patient.

# Frontend

## New Features

### Patient Notes Interface

A dedicated interface has been implemented for managing independent patient notes:

- Separate view decoupled from the medical record module.
- Pagination support.
- Optimized content visualization.
- Default text truncation with expansion on user interaction.

### "Background" Field in Medical Record

The `Background` field has been added to the Medical Record view, corresponding to the new `background` backend attribute.

- Allows viewing and editing general medical background.
- Integrated into reactive forms.
- Included in PDF export.

---

## Improvements and Refactoring

Due to increased complexity, the patient detail modal has been extracted into its own component:

- Better internal routing organization.
- Increased maintainability and Angular code scalability.

  - Proper centering
  - Appropriate background handling
